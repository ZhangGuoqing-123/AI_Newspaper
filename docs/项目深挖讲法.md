# 硅谷速递 · AI 产品经理面试讲法手册（项目深挖）

> 用途：简历关 + 面试"项目深挖"。核心目标——讲出 **agent（非 workflow）+ 多模态 AIGC（非生文）+ 业务×技术冲突**。
> 配套理念见知识库笔记《AI产品经理 项目深挖》：面试官挖的是技术边界感、业务权衡力、真实泥沼里的思考路径。

---

## 0. 一句话定位

> 一个**自主 agent**：把 Twitter 上嘈杂的 AI 资讯，按用户选定的信源，自动完成
> **选题 → 多模态生成（封面图 + 个性化音频 + 数字人短视频）→ 质量自检不达标就重做**，
> 产出"可读、可听"的个性化日报，并用数字人短视频做获客分发。

---

## 1. 升级动机（为什么改）

- **改造前（被刷的版本）**：爬 Twitter（TikHub）→ Kimi K2.5 出文字摘要。两步写死 = **workflow + 生文**，技术含量低。
- **市场信号**：AI PM 实习不要纯 workflow，要 **agent**；AIGC 不要生文，要 **多模态（生图/生视频）**。
- **改造后**：把线性流程升级为**有自主决策 + 自检闭环的 agent**，产出**多模态**内容。

---

## 2. 系统架构（agent + 多模态流水线）

```
       ┌────── 编排逻辑(代码)驱动 agent，在决策点调 LLM ──────┐
[工具:TikHub爬取] → 预过滤(embedding去重/水贴过滤) → [LLM]选题排序
                  → [Kimi K2.5]写结构化摘要（实测多模型后选定的摘要模型）
                  → 为选中故事配图，调【生图: Nano Banana】生成封面      ← 多模态:生图
                  → 头条改写口播稿，调【TTS】出音频 + 调【数字人:LipSync/Seedance】出视频  ← 多模态:生视频
                  → [质检: LLM-as-judge] 摘要忠于原文吗/口播稿顺不顺/图配得对不对？
                        └─ 不达标 → 自动重生成（最多 N 次）                ← 这一步让它够得上 agent
                  → 组装：按用户信源拼装个性化日报；统一短视频用于分发
```

**模型分工（按任务选最好的，不是一个模型包打）**：摘要节点用 **Kimi K2.5**（拿真实数据实测多个模型后选定）；选题排序、口播稿改写、质检等决策节点用 LLM（见第 10 节，v1 可同用一个，后续按需优化）。
**它凭什么算 agent（而非 workflow）**：有**自主决策**（选题、配图与否、质量是否过关）、**工具调用**（爬取/生图/TTS/生视频）、**反思重试循环**，由编排逻辑指挥，不是写死的 A→B→C。

---

## 3. 三个核心设计决策（讲法）

### 3.1 个性化分工：消费功能个性化，分发功能统一化
| 模态 | 个性化? | 角色 |
|---|---|---|
| 文字摘要 | 个性化（绑用户信源）| 消费 / 留存 |
| 音频播报 | 个性化（通勤听你的信源）| 消费 / **变现** |
| 数字人短视频 | **统一**（全平台 top 故事）| **获客分发** |

### 3.2 数字人到底解决什么需求（关键反转）
- 诚实结论：让数字人念新闻，**对用户消费没价值**（看脸不增加信息）。
- 真实定位：**它是获客/分发引擎，不是消费功能**。竖屏短视频是抖音/视频号原生货币，零边际成本天天产，铺到站外**拉新**。
- 讲法："音频解决用户通勤消费，数字人短视频解决**我平台的获客分发**，两个模态对应两类不同需求。"

### 3.3 多模态选型：能可控就别用生成式
- 数字人念的是 Kimi 写好的**逐字稿**（可控），不是让模型自由生成画面，内容百分百对得上文案；封面图这种**装饰性**的才用生成图。
- 讲法："能用可控的地方就不用生成式，把不可控留给无关紧要的装饰。"

## 4. "业务×技术碰撞"（面试深挖核心，只留真实的）

> 原则：宁缺毋滥。下面两个是**这个项目里真实存在、且能体现技术栈/架构思考**的碰撞，不硬套。
> 已删除原"可信合规×生成幻觉"那条——硬套催收机器人：数字人念的是逐字稿，不存在 text-to-video 幻觉，合规只是 checkbox，不构成架构碰撞。**延迟也不是冲突**（批量推送、夜间预生成即可），面试别讲延迟。

### 碰撞① 业务要"高信噪比简报" × 技术"洪流没法低成本可靠地自动选题"（agent 核心难点）
| 维度 | 内容 |
|---|---|
| 业务诉求 | 简报要高信噪比：去重、去水贴、按重要性排序——本是人工主编的活 |
| 技术极限 | TikHub 一天抓回上千条，**全塞进 Kimi 不现实**（上下文塞不下、成本高、判断不稳）|
| 我的取舍 | 先用 **embedding 相似度去重 + 互动量/信源可信度预过滤**把上千条砍成候选集 → 再让 **Kimi 在 shortlist 上选题、排序**。即"**便宜的过滤在前、贵的 LLM 判断在后**" |
| 体现的思考 | embedding/检索、上下文与成本经济、agent 的选题决策链 |
| 量化指标 | 候选从 __ 条压到 __ 条；选题准确率（人工抽检）__%（待实测）|

### 碰撞② 业务要"千人千面个性化" × 技术/经济"个性化内容没法跨用户复用"（架构核心）
| 维度 | 内容 |
|---|---|
| 业务诉求 | 每用户绑自己信源，**个性化音频**专属 = 差异化卖点 |
| 技术极限 | 内容每人独一份 → 音频没法跨用户缓存 → TTS 成本随**用户数线性增长** |
| 我的取舍 | **按"故事"缓存音频片段，而非按"用户"**：同一条推文出现在很多人简报里，TTS 只生成一次，谁有这条就复用；个性化只发生在"选哪些、怎么排序、怎么拼装"。→ 成本从 `O(用户数×故事数)` 降到 `O(去重故事数)`（有上限）|
| 体现的思考 | 缓存键设计、把生成成本与用户数解耦、生成流水线架构 |
| 量化指标 | 故事级缓存命中率 __%、单用户日均 TTS 成本从 ¥__ 降到 ¥__（待实测）|

> 注：数字人视频是**统一**的（全平台一条做分发），本就不存在按用户爆炸的问题——成本碰撞集中在**个性化音频**，别夸大成"生图+生视频都爆炸"。

## 5. 商业模式

- **内容不直接赚钱**，承担获客与数据沉淀。
- **数字人短视频**：免费，铺站外（抖音/视频号）做**获客**。
- **个性化音频**：**付费**，卖给有**通勤刚需**的人，用于**回本**（承认是回本，不是利润引擎——诚实）。
- **真正的变现（roadmap / What-if）**：用户量起来后，把沉淀的 **AI 赛道趋势 + 用户关注行为数据**做成**投研/线索数据产品**，卖给**中小私募**。
  - **数据壁垒（必须能答）**：聚合 Twitter **一手、早于主流数据库**的 AI 信号 + 平台关注热度 → "早期趋势 + 热度" alpha；私募买的是"**比 PitchBook 更早的苗头**"。
  - 是**单边数据生意**（卖数据给私募），**不是**双边"找投资人"撮合（避开双边冷启动）。
  - 坦诚："会先验证数据壁垒、再投入 B 端销售/合规。"

## 6. 合规（别漏）

国内《深度合成规定》：AIGC 内容需**显著标识 + 内容审核**。架构里有专门的标识与审核节点。——主动提合规 = 成熟度信号。

## 9. 预期追问 & 标准答法

- **Q：你这凭什么叫 agent，不是 workflow？**
  A：有自主决策点（选题、配图与否、质量是否过关）+ 工具调用 + 自检重试循环，是 LLM 在指挥流程，不是我写死流程。
- **Q：数字人口播解决什么需求？**
  A：不是消费功能，是获客分发引擎（站外短视频拉新），零边际成本。
- **Q：免费视频会不会把付费吃掉？**
  A：免费视频是**通用**内容（全网 top），付费是**个性化**（你的信源）。只满足通用头条的人本就低付费意愿，正好做传播量；有具体信息需求的人通用视频喂不饱，必须订阅。免费/付费服务不同用户意图，设计上防蚕食。
- **Q：上千条推文你怎么自动选出该报的？**
  A：embedding 去重 + 互动量/信源可信度预过滤砍成候选集，再让 Kimi 在 shortlist 上选题排序——便宜的过滤在前、贵的 LLM 判断在后（碰撞①）。
- **Q：个性化音频成本怎么控制？**
  A：按"故事"缓存 TTS 片段（不按用户），成本与用户数解耦（碰撞②）。
- **Q：生成的图/视频会不会内容失真？**
  A：数字人念逐字稿（可控），封面图只是装饰，不靠自由生成承载事实；AIGC 内容做合规标识。
- **Q：怎么衡量做得好不好？**
  A：技术侧——事实自检通过率、重生成率、缓存命中率；业务侧——音频完播率、付费转化、短视频带来的新增。
- **Q：为什么用 Kimi K2.5？**
  A：我拿真实爬取的推文**实测了多个模型**，Kimi 在**中文摘要质量**上最好，所以**摘要节点**选它。它不是 agent 的大脑——编排和决策是分开的（按任务选模型）。
- **Q：多少用户 / 收入？**
  A：还没开始做用增；这是**按商用标准做的产品原型**，用来逼出真实的工程取舍，商业化是规划（见第 5 节）。

## 10. 技术栈 & 待实测指标清单

| 环节 | 选型 | 备注 |
|---|---|---|
| 爬取 | TikHub | 现仅 Twitter，规划多平台 |
| 编排/决策（选题/排序/质检）| LLM（v1 待定，可同用一个）| agent 的"自主"靠它+编排逻辑 |
| 摘要 | **Kimi K2.5**（实测多模型后选定）| 中文摘要质量最好 |
| 生图 | Nano Banana（DreamAPI）| 故事级封面 |
| 语音 | TTS（中文自然度优先：火山/MiniMax 等）| 故事级缓存、按用户拼装 |
| 数字人视频 | LipSync / Seedance（DreamAPI）| 统一短视频，供分发 |
| 质检 | LLM-as-judge | 事实/质量打分 + 重生成 |

**待 demo 跑出来填的真实数字**：单条生图成本、单条视频成本/耗时、单用户日均生成成本、故事级缓存命中率、事实自检通过率、重生成率。
